# Resources System — Complete System Design

## Purpose

A personal resources system that:

- Captures durable facts from multiple sources: Confluence, web, files, meeting notes, journal entries.
- Organizes knowledge as an atomic concept graph: one article per concept, cross-linked.
- Stays healthy automatically: all mechanical checks run as JavaScript scripts.
- Compounds over time: every ingest, query, enrichment, and discovery pass makes it more valuable.
- Integrates with Confluence on a regular cadence: new and modified pages flow into resources automatically.

---

## Design Principles

1. **Script everything mechanical.** If a check is deterministic (count lines, parse YAML, find broken links, compare files to a table), it is a JavaScript script. The LLM never runs `rg`, `grep`, or `find` inline.
2. **LLM handles only semantic work.** Enrichment, synthesis, ingest decisions, query answering, structural planning — anything requiring judgment or writing goes through a skill workflow.
3. **Scripts produce structured output; LLM consumes it.** Every script writes to stdout in a consistent format (checklist, table, or plain list). The LLM reads the output and acts on it.
4. **Scripts never call the LLM** — except `run-operation.js` and `plan-resources.js`, which invoke `opencode run` as a deliberate subprocess. That boundary is explicit and documented.
5. **Confluence is raw material.** Resources is not a mirror of Confluence. It distills durable facts. One page may feed multiple articles; multiple pages may feed one.
6. **The graph is the asset.** Cross-references build value. Atomic articles linked densely are worth more than long disconnected documents.

---

## Architecture

Four layers. Data flows downward; LLM reads results upward.

```
Commands  →  Skills  →  Scripts  →  Storage
```

| Layer | What it is | Who runs it |
|---|---|---|
| Commands | Slash commands — entry points for the user | User invokes; LLM executes |
| Skills | Markdown workflow files — LLM procedures | LLM follows |
| Scripts | JavaScript files — mechanical operations | LLM invokes via `node` |
| Storage | Markdown files — the data | Scripts and LLM read/write |

## Storage Layer

### Resource Articles — `resources/`

One file per concept. Atomic: each article covers exactly one thing. Every article has:

- YAML front matter: `title`, `status`, `updated`, `actualized`, `tags`.
- An Overview section with real content (no stubs).
- `## Related` — bidirectional cross-references to linked articles.
- `## Sources` — external sources used to build the article.
- `## Changelog` — append-only edit history.

Subfolders are concept domains (defined in `context.md`). Special subfolders:

| Subfolder | Contents |
|---|---|
| `resources/people/` | One article per person. Exempt from orphan check. |
| `resources/teams/` | One article per team. |
| `resources/tools/` | Developer tools and internal tooling. |
| `resources/adr/` | Architecture Decision Records. |

### Journal — `journal/`

| Path | Contents |
|---|---|
| `journal/daily/YYYY-MM-DD.md` | Daily notes — append-only |
| `journal/weekly/YYYY-WNN.md` | Weekly notes |
| `journal/meetings/YYYY-MM-DD-<slug>.md` | Meeting notes |

### Inbox — `inbox/`

| Path | Contents |
|---|---|
| `inbox/scratch.md` | Running scratch pad — unprocessed captures |

### Index Files

| File | Purpose | Updated by |
|---|---|---|
| `knowledge-graph.md` | Master index of all resource articles: file, summary, tags | LLM after every article create/update/move/delete |
| `tags.md` | Tag registry — all registered tags with descriptions | LLM when new tag needed |
| `confluence-map.md` | Confluence page index: page ID, title, parent, last modified, summary, tags | LLM + `confluence-backfill-dates.js` |
| `resources-log.md` | Append-only operation log: ingest, query, lint, sync entries | LLM appends after each operation |
| `resources-actualize-plan.md` | Current operation queue from Workflow C or B | Generated by `plan-resources.js` or LLM directly |

## Scripts Layer

All scripts: JavaScript, run via `node scripts/<name>.js [args]` from repo root.
All scripts write to stdout. The LLM reads stdout and acts on it. Scripts never write to repo files except where noted.

### Shared Library

| File | Exports |
|---|---|
| `scripts/lib/links.js` | `findMdFiles(root)`, `extractLinks(content)`, `resolveLink(from, to)`, `findBacklinks(file, root)` |
| `scripts/lib/frontmatter.js` | `parseFrontmatter(content)`, `hasFrontmatter(content)`, `getFrontmatterField(content, field)` |
| `scripts/lib/graph.js` | `readKnowledgeGraph(path)`, `getGraphEntry(file)`, `listResourceFiles(root)` |
| `scripts/config.js` | `CONFLUENCE_BASE`, `CONFLUENCE_SPACES`, `authHeader()`, `loadMapIds()`, `REPO_ROOT` |

### Health Scripts

Run before/after any batch operation, on `/lint`, or on demand. All output to stdout as a checklist.

| Script | Purpose | Output |
|---|---|---|
| `health-frontmatter.js` | Detect `.md` files missing `updated` or `tags` front matter fields (all files except AGENTS.md) | Checklist of violations |
| `health-links.js [--scope resources\|journal\|all]` | Detect broken relative links and missing back-links in `resources/` (and optionally `journal/`) | Checklist of broken links; list of missing back-links |
| `health-orphans.js` | Detect articles in `resources/` with zero inbound links from other resource files. People files exempt. | List of orphan paths |
| `health-kg-coverage.js` | Compare every file in `resources/` to rows in `knowledge-graph.md`. Report files not in graph. | List of uncovered paths |
| `health-tags.js` | (a) Tags used in front matter not registered in `tags.md`. (b) Tags registered in `tags.md` not used anywhere. | Two lists: orphan tags, dead registrations |
| `health-stale.js [--days N]` | Articles where `actualized` date is older than N days (default: 90). Falls back to `updated` if `actualized` absent. | List of stale paths with days since actualization |
| `health-lengths.js [--lines N]` | Articles in `resources/` with body content over N lines (default: 80). Split candidates. | List of long articles with line counts |
| `health-projects.js` | Active projects (`status: Active`) with no dev log entry in last 30 days, or all tasks completed. | List of stale projects |
| `health-sections.js` | Standard sections (`## Log`, `## Tasks`, `## Notes`, `## Dev Log`) that exist but have no content. | List of empty sections |
| `health-all.js` | Run all health scripts in sequence. Emit unified checklist grouped by check type. | Full lint report |

### Discovery Scripts

| Script | Purpose | Output |
|---|---|---|
| `discover-connections.js [--scope <path>]` | For all articles in scope: compute backlinks, shared tags, shared sources, entity co-occurrence, structural proximity. Score pairs. | Scored pair table: Article A, Article B, Score, Signals |

### Analysis Scripts

| Script | Purpose | Output |
|---|---|---|
| `report-resources.js [--sort actualized\|lines\|inlinks]` | For every article in `resources/`: path, line count, tag count, actualized date, inbound link count, outbound link count. | Structured table; highlights split candidates and stale articles |

### Confluence Scripts

| Script | Invocation | Purpose | Output |
|---|---|---|---|
| `confluence-updates.js` | `node scripts/confluence-updates.js --since YYYY-MM-DD` | Fetch pages modified since date in configured spaces. Report which are in/out of `confluence-map.md`. | Table: page ID, title, in-map status, last modified |
| `confluence-missing.js` | `node scripts/confluence-missing.js` | Enumerate all pages in configured spaces. Report which are missing from `confluence-map.md`. | Table: page ID, title, space, missing flag |
| `confluence-backfill-dates.js` | `node scripts/confluence-backfill-dates.js` | Call Confluence REST API for each row in `confluence-map.md`. Write `version.when` dates in place. | Progress log; count of rows updated |

### Ingest Scripts

| Script | Invocation | Purpose | Output |
|---|---|---|---|
| `fetch-source.js` | `node scripts/fetch-source.js <url>` | Fetch URL, strip navigation/ads/boilerplate, emit clean markdown. | Clean markdown to stdout |

### Operation Scripts

These scripts invoke `opencode run` as a subprocess.

| Script | Invocation | Purpose |
|---|---|---|
| `fix-links.js` | `node scripts/fix-links.js` | Auto-fix missing back-links in `## Related` sections across `resources/`. Writes to files. |
| `run-operation.js` | `node scripts/run-operation.js --op <type> --file <path>` | Execute one structural operation (delete/merge/split/create/reclassify/actualize) via `opencode run`. |
| `plan-resources.js` | `node scripts/plan-resources.js` | Generate `resources-actualize-plan.md` via `opencode run` with Workflow C prompt. |

### Search Scripts

| Script | Invocation | Purpose |
|---|---|---|
| `qmd-index.js` | `node scripts/qmd-index.js` | Re-index qmd after batch ingest or enrichment. Collections: `resources/`, `journal/`, `inbox/`. |

qmd query runs directly via CLI:
```bash
qmd query "<question>"
```

## Skills Layer

Skills are LLM workflow files. Each skill defines a workflow the LLM executes step by step.

### `resources/SKILL.md` — Core

Loads alongside all resource sub-skills. Contains:
- Resources philosophy (concept graph, one concept per article, freely restructure).
- Dispatch table: which sub-skill to load for each task.
- Rules (bidirectional links, no stubs, distill don't copy, etc.).

Does **not** contain workflow procedures — those live in sub-skill files.

### `resources/operations.md` — Workflow D: Structural Operations (EXTRACTED)

Currently embedded in `SKILL.md`. Move to its own file for consistency with all other workflows.

Contains the five primitive operations: delete, merge, split, create, reclassify. All other workflows (A, B, F) delegate to these procedures.

Load when: executing any structural operation on the graph. All workflows that perform structural ops must load this sub-skill first.

### `resources/enrich.md` — Workflow A: Actualize Single Article

**Input:** article path.

**Steps:** Read → Gather data (local resources, Confluence, Jira, codebase, journal) → Enrich → Remove stale content → Fix cross-references → Update metadata → Health check → Commit.

**Health check step (Step 7)** calls scripts:
- `node scripts/health-orphans.js` — check if this article is an orphan; fix if so.
- `node scripts/health-tags.js` — verify tags are consistent.
- `node scripts/health-stale.js --days 14` — spot-check nearby articles for staleness.

**One article per invocation. Stop after committing.**

### `resources/sync.md` — Workflow B: Confluence Sync

**Input:** none. Operates on all configured spaces.

**Steps:**
1. `node scripts/confluence-missing.js` → detect missing pages.
2. Add missing pages to `confluence-map.md` (batches of 10).
3. `node scripts/confluence-updates.js --since <last-changelog-date>` → detect modified pages.
4. Update modified rows.
5. `node scripts/confluence-backfill-dates.js` → fill Last Modified dates.
6. Add changelog entry to `confluence-map.md`.
7. Produce operation plan (same format as Workflow C Step 4 — inline, not via script).
8. Execute operations following Workflow D + Workflow A rules.
9. `node scripts/health-all.js` → graph health check. Report issues in commit message.
10. Commit.

### `resources/plan.md` — Workflow C: Graph Review

**Input:** none. Operates on all resources.

**Steps:**
1. `node scripts/report-resources.js` → get full article inventory (sizes, dates, links).
2. Read `knowledge-graph.md`, `tags.md`, `confluence-map.md`.
3. Read all articles (using report as a guide for priority).
4. `node scripts/discover-connections.js` → get connection signals.
5. Identify graph problems: merges, splits, new nodes, deletions, reclassifications, missing cross-references.
6. Write `resources-actualize-plan.md` — ordered operation queue.
7. Commit.

**Planning only — do not edit any resource articles.**

### `resources/ingest.md` — Workflow F: Ingest Source (NEW)

**Input:** URL, local file path, or pasted content.

**Steps:**
1. Fetch content:
   - URL: `node scripts/fetch-source.js <url>` → clean markdown.
   - Local file: read directly.
   - Paste: use as-is.
2. Identify concepts: what topics, entities, decisions does this source contain?
3. For each substantial concept: check `knowledge-graph.md`. Decide: `create` (no article exists) or `actualize` (article exists, needs new facts).
4. Execute decisions following Workflow D (create) or Workflow A (actualize).
5. Update `knowledge-graph.md` for new articles.
6. `node scripts/qmd-index.js` → re-index.
7. Append to `resources-log.md`:
   ```
   ## [YYYY-MM-DD] ingest | <source title or URL>
   Source: <url or path>
   Operations: <list of create/actualize ops performed>
   ```
8. Commit.

### `resources/query.md` — Workflow G: Journal Query (NEW)

**Input:** a question.

**Steps:**
1. `qmd query "<question>"` → get top matching articles.
2. Read the returned articles in full.
3. Synthesize an answer with citations (links to source articles).
4. Assess: is this answer worth persisting in resources? File it if:
   - It synthesizes 3+ articles on a theme not yet captured.
   - It connects concepts in a way no existing article does.
   - It answers a question likely to recur.
5. If filing: create a new article in `resources/synthesis/` following Workflow D Create. Set `status: synthesis`.
6. Append to `resources-log.md`:
   ```
   ## [YYYY-MM-DD] query | <question summary>
   Filed: <path> (or "not filed")
   ```

### `resources/discover.md` — Workflow E: Discover Connections

**Input:** scope (single article, tag cluster, subfolder, or full graph).

**Steps:**
1. `node scripts/discover-connections.js [--scope <path>]` → get scored connection pairs.
2. Filter to score ≥ 3.
3. Generate connections report: missing cross-references, isolated articles, cluster map.
4. If user approves: patch articles with new `## Related` links, commit.

## Commands Layer

| Command | Arguments | Workflow | Purpose |
|---|---|---|---|
| `/r-enrich` | `<article-path>` | Workflow A | Actualize one article from all sources |
| `/r-sync` | none | Workflow B | Full Confluence sync → resource operations |
| `/r-plan` | none | Workflow C | Graph review → operation plan |
| `/r-ingest` | `<url-or-file>` | Workflow F | Ingest arbitrary source into resources |
| `/ask` | `<question>` | Workflow G | Query all journal data; optionally file result to resources |
| `/r-lint` | none | `health-all.js` | Run all health checks; report violations |
| `/r-discover` | `[scope]` | Workflow E | Discover hidden connections |

### `/r-lint` implementation

```
1. node scripts/health-all.js
2. Present output as grouped checklist to user.
3. Append to resources-log.md:
   ## [YYYY-MM-DD] lint
   Issues: <count> | Clean: <checks with zero issues>
4. No files modified — read-only audit.
```

## Confluence Integration

Confluence feeds resources through a two-phase regular cadence:

**Phase 1 — Sync** (`/r-sync`):
- `confluence-missing.js` finds new pages; LLM adds them to `confluence-map.md`.
- `confluence-updates.js --since <date>` finds modified pages; LLM updates map rows.
- `confluence-backfill-dates.js` fills Last Modified dates.
- LLM produces an operation plan from actionable pages (content with durable facts).
- LLM executes: delete → merge → reclassify → split → create → actualize.

**Phase 2 — Enrich on demand** (`/r-enrich`):
- Individual articles are enriched using Confluence as one of five sources.
- The plan file (`resources-actualize-plan.md`) from Phase 1 drives what to enrich next.

**Regular cadence:** Run `/r-sync` weekly or after significant Confluence activity. Run `/r-enrich` on individual articles between syncs as needed.

## Search — qmd

qmd provides hybrid BM25 + dense vector + reranking search over local markdown.

**Config:** `qmd.yml` at repo root.

```yaml
collections:
  - name: resources
    path: ./resources
    glob: "**/*.md"
  - name: journal
    path: ./journal
    glob: "**/*.md"
  - name: inbox
    path: ./inbox
    glob: "**/*.md"
```

**Install:** `npm install -g @tobilu/qmd` (requires Node.js ≥ 22; downloads ~2GB models on first run).

**LLM usage:**
```bash
# Search across all collections
qmd query "<question>"

# Re-index after batch changes
node scripts/qmd-index.js
```

**When to re-index:** after `/ingest`, after `/resources-sync`, after any batch of enrichments.

## `resources-log.md` Format

Append-only. New entries prepended (newest first within file, but entries are appended — never edited).

```markdown
---
type: meta
updated: YYYY-MM-DD
tags: []
---

# Resources Log

## [YYYY-MM-DD] lint
Issues: N (frontmatter: A, links: B, orphans: C, tags: D, stale: E, kg-coverage: F)

## [YYYY-MM-DD] ingest | <source title or URL>
Source: <url or file path>
Operations: create resources/domain/concept.md, actualize resources/domain/other.md

## [YYYY-MM-DD] query | <question summary>
Filed: resources/synthesis/answer-slug.md
(or: not filed)

## [YYYY-MM-DD] sync
Pages added: N, modified: M
Ops: D deleted, G merged, C created, U actualized
```

## Constraints

- All scripts: JavaScript, run via `node`. No shell scripts, no Python, no inline `rg`/`grep` in skills.
- Scripts write to stdout only — except `fix-links.js` (writes to files), `confluence-backfill-dates.js` (writes to `confluence-map.md`), `plan-resources.js` (writes `resources-actualize-plan.md` via `opencode run`), and `run-operation.js` (writes to files via `opencode run`).
- Never write to Confluence or Jira without explicit user approval.
- qmd requires Node.js ≥ 22 and ~2GB model download on first run.
- `context.md` drives Confluence space config. When Confluence is not configured, sync workflows degrade gracefully (scripts report no credentials and exit cleanly).
- The LLM reads all script output and decides what to do — scripts never make editorial decisions.
